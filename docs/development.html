1. Hosting Setup

The web application is hosted on the FH St. Pölten Campus Cloud using a Node.js instance. The project is fully functional and accessible online. (Not anymore)

Tools & Services

  • Node.js backend (Express)
  • MySQL database hosted via FH St. Pölten
  • React frontend (Vite-based setup)
  • Campus Cloud FTP access (FileZilla)

Deployment Process

  • Built the frontend using npm run build (created dist/)
  • Created /frontend and /backend folders inside the Campus Cloud /node directory
  • Uploaded backend files (excluding node_modules) and frontend dist/
  • Updated app.js to serve the static frontend
  • Adjusted all Axios requests to use the /api prefix
  • Final startup command: node app.js

3. Code Documentation

The codebase is structured and commented to remain understandable and maintainable.

  • Clear and consistent variable naming (e.g. resonateCount, isAdmin)
  • Each file has a focused responsibility
  • Controllers handle business logic
  • Routes define API endpoints
  • Models manage database access
  • Frontend split into pages/ and components/
  • Complex functions include inline comments
  • Reusable CSS classes across components

5. User Interaction Overview

Dear Body is a poetic blog-based web application where users can write anonymous letters to their body or body parts as a form of reflection and emotional release.

User Flow

  • Landing page → read or write letters
  • Reading letters is public
  • Writing and resonating requires login
  • Letters are stored in the database
  • All letters appear anonymous to other users
  • Logged-in users can view their own letters

Database Design

  • usersDearBody: id, username, email, password_hash
  • lettersDearBody: id, user_id, title, body_part, content, created_at, resonate_count

6. MVP Scope

Frontend

  • Login and registration flow
  • Form to write letters (auth-protected)
  • Public feed of letters
  • Profile page with user’s own letters

Backend

  • Authentication endpoints
  • CRUD functionality for letters
  • Protected routes using middleware

Database

  • Two core tables: users and letters

Known Issues & Limitations

  • No pagination (letters are listed continuously)
  • No edit/delete functionality for user letters
  • No advanced accessibility features implemented

Overall, I am very satisfied with the final result. While it does not fully match my initial design vision, this was a conscious trade-off in favour of functionality and completion.
